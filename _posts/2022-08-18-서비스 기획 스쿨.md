# 서비스 기획 스쿨
## Chapter 3
### 마인드맵을 활용한 시스템 기획법
* 마인드맵 3단계  
  1) 기능정의 및 데이터 정의
  2) 사용자별 플로우 정의(제휴 마케팅 어드민 / 사용자)
  3) 간단하게 개발대상 페이지 구분하기  


* 네트워크를 활용한 모든 온라인 서비스는 결국 '기능' 과 이 기능을 타고 움직이는 '데이터' 로 움직인다.


* 예시) '인근 미용실 지도'  
    - 데이터: 미용실 상호, 미용실 로고, 미용실 주소지  
    - 기능: 데이터를 지도에 표시해주는 것.  
            ex) 지도 위에 어떤 방식으로 표시해 줄지 하는 것, 지도 위의 표시를 눌렀을 때 상세한 정보(데이터)를 보여주는 것.  
            
            
* 디폴트 설정은 논리적인 사유로.  
    - '그냥 반경 ㅇㅇ미터가 좋겠다.' 가 아니라, '10분 내외로 걸을 수 있는 200m 정도로 잡는다.'  


* 사용자를 파악한다는 것은 사용자의 문화인류학적 정보인 연령이나 가족관계가 아니라 사용자가 서비스에서 이루려는 목적이나 상황을 인지하여 적절한 프로세스를 설계해주는 것이다.  


### 기획자의 실수 1) 목적에 맞는 범위로 좁힌 것이 맞나?
* 현재 운영 중인 서비스에 새로운 서비스를 만드는 일은 이미 다 만들어진 시스템에서 기능의 일부를 바꾸거나 추가하는 것을 의미한다.  
  따라서, 개선을 위해 기존 기능 그대로 두어도 되는 것은 냅두고, 바꾸거나 추가할 기능을 만들기 위한 케이스와 운영 방식을 구체화 시켜야 한다.  
  
  
* 연구 가설이 옳았는지 보려면 전략적으로 바꾸려고 하는 부분 외에 나머지 부분은 모든 조건이 동일해야 한다.  
  관계없는 개선사항이 끼어드는 순간, 오픈 후 서비스가 정말 전략대로 수정되었는 지 확인할 수가 없다.  
  
  
### 실수 2) 한 가지 케이스로만 만들지 않았나?
* 템플릿: 서비스의 모든 기능은 어떤 데이터가 들어와도 동일한 기준으로 문제없이 작동할 수 있어야 하는데, 이러한 개념을 통틀어 이르는 말.  


* 서비스 기획에서 중요한 것은 들어올 가능성이 있는 모든 데이터에 대해 케이스별로 적절한 처리 방식을 예상해주는 것이며,  
  예상하지 못한 데이터가 들어오더라도 서비스가 문제되지 않게끔 미리 해결하는 것.  
  

* 네이버와 다나와를 기준으로 기획할 것이 아니라 제휴 진입의 두 가지 데이터 유형을 구분하여(Soft유형/Hard유형) 그에 따라 작동할 수 있게 만들어야 한다.  
  그렇게 만들면 제휴처럼 수십, 수백, 수천 개가 되어도 두 가지 유형 중 하나에 포함될 것이고, 서비스는 템플릿으로서 어떤 기능을 제공해야 할지 정의할 수 있게 된다.   


### UI 설계를 해보자  
* 마이크로 UX: 케이스별로 UI가 다르게 관리되어 사용자가 편리하게 사용할 수 있도록 유도하는 설계.  
  ex) 회원가입을 할 때, ID를 입력하고 존재 여부를 검색할 때    
      1) 이미 존재하는 ID인지
      2) 아니면 등록 가능한 ID인지
      3) 그도 아니면 ID 등록 규정에 따르지 않는 것인지
      에 따라서 피드백이 모두 다르게 나오게 하는 것.  
      
* 1) 스케치: 서비스의 핵심적인 부분만을 집중적으로 고민해볼 수 있다.  
            디테일한 UI를 그리기보다는 여러 가지 UI 형태를 배치해보는 게 중요.  
            
  2) 와이어프레임: 네모와 텍스트 정도의 간단한 조합으로 정리된 영역들을 합쳐 페이지 전체의 구성을 볼 수 있도록 한 것.  
                  스케치를 좀 더 구체화하기 위해 영역과 구성요소들을 단순하게 표현해 디지털 문서화시키는 경우가 많음. 
                  사용툴: 파워포인트-PowerMockup, Axure의 목업툴, 크롬-draw.io(more shapes)
                  
  3) 목업: 화면 UI가 2D 형태로 만들어져 있는 형태.  
           겉으로는 다 만들어진 것 같지만 인터랙션이나 기능적인 움직임이 전혀 없는 상태.  
           
  4) 아트보드: 하나의 화면이 다양한 해상도에서 어떻게 보이는지 나란히 펼쳐놓거나 하나의 서비스에 포함된 화면들을 순서대로 주욱 펼쳐놓은 형태.  

  5) 프로토타입: 동적. 프로세스를 검증하거나 UI의 구성과 위치를 검증하는 단계  
                사용툴: 카카오-oven

* 페이퍼 목업 테스트  


### 화면설계서와 사용자 스토리  
* 화면설계서 툴 : 액슈어Axure, 스케치Sketch  


* 좋은 사용자 스토리 조건 6가지  
  1) 독립적이다.
  2) 협상 가능하다.
  3) 사용자와 비즈니스에 가치가 있다.
  4) 예측과 측정이 가능하다.
  5) 사이즈가 작아야 한다(2~3주내 개발 가능).
  6) 테스트가 가능해야 한다.  
  
  
* 에픽: 특정 사용자 스토리에서 최상단에 놓이는 사용자 스토리.  
        제휴고객에게 앱으로 유도할 추천상품을 보여주는 것까지도 굉장히 많은 로직이 필요하다.  
        일단 제휴로 들어와 주문한 고객이라는 것을 구분하는 것도 하나의 기준이 되고, 추천상품을 노출할 때 어떤 기준으로 노출할 것인지에 대한 부분도 있을 수 있음.  
        이렇게 특정 사용자 스토리가 너무 많은 기능을 포함하고 있을 때는 이를 가장 상위에 놓고(에픽) 그 하위(백로그)에 들어갈 사용자 스토리를 모으거나 새로 작성해 체계를 잡을 수 있다.  
        ex) 기준이 되는 사용자 스토리(에픽): '회원의 상품 주문'  
            하위 사용자 스토리: '신용카드로 결제하기', '간편페이로 결제하기'  
            
* 백로그: 하위의 가장 작은 기능 단위로 정의된 항목  

* 칸반 보드: 해야 할 일, 하고 있는 일, 완료한 일로 영역을 구분하고 해당 스프린트에 포함된 백로그를 진행 상태에 따라 이동시킨다.
